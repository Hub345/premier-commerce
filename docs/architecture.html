§01 Decisions on record

Six calls that shape everything downstream.

Each was a fork with a real trade-off. These are the ones we locked before writing code — the rationale in one line so the next person on the team inherits the why, not just the what.

Multi-tenancy
Design for 500, build for 1

Every table is tenant-ready from row one. The provisioning UI, custom domains, and theming engine wait until tenant two signs — no machinery serving zero customers.

Checkout
Guest, phone-first

Buy with a phone number and M-Pesa — no account required. Optional accounts are offered after the sale, where they cost no conversion.

Payments
Bizrah’s own Till

Money settles directly to the merchant. Each tenant brings its own Daraja credentials, so there’s no float, KYC, or aggregator licensing on the platform.

API
One versioned contract

A thin BFF exposes v1. The web store today and the Flutter admin app tomorrow consume the exact same surface — logic lives in one place.

Variants
First-class from day one

SKU, stock, price, images, and attributes live on the variant, not the product. A 256 GB phone and its 128 GB sibling are distinct sellable units.

Search
Portable by design

Postgres full-text now, behind a provider interface. Swapping in Meilisearch or Algolia later touches one adapter — never the frontend.

The load-bearing rule

The client never sends business_id. It is derived from the request’s host, server-side, and stamped into context before a single query runs.

That one discipline closes the most common multi-tenant leak — a client asking for another tenant’s data and being believed.

-- Stock, price and images live on the variant, not the product.
create table product_variants (
  id          uuid primary key default gen_random_uuid(),
  business_id uuid not null references businesses(id) on delete cascade,
  product_id  uuid not null references products(id)   on delete cascade,
  sku         text,
  label       text,                        -- "256GB / Graphite"
  price_cents bigint not null,               -- integer money, never floats
  stock       integer not null default 0,
  attributes  jsonb not null default '{}',   -- storage, color, warranty…
  images      text[] not null default '{}'
);

-- Public reads only published rows; writes require membership.
create policy variants_public_read on product_variants
  for select using (exists (
    select 1 from products p
    where p.id = product_variants.product_id
      and p.status = 'published'));

§04 Standard Commerce Protocol

One typed contract. Web today, Flutter tomorrow.

The storefront and the future admin app never talk to the database directly — they call this surface. It is the seam that keeps the frontend decoupled from Postgres and lets a Dart client consume the identical API a year from now.

packages/protocol/src/client.ts
export interface CommerceClient {
  catalog: {
    listProducts(q: ProductQuery): Promise<Paginated<Product>>;
    getProduct(slug: string):    Promise<Product>;
    listCategories():             Promise<Category[]>;
    search(term: string, f?: ProductFilters): Promise<Product[]>;
  };
  checkout: {
    // Server reprices from the DB and computes VAT — client totals are never trusted.
    createOrder(input: CreateOrderInput): Promise<Order>;
  };
  payments: {
    initiateMpesa(orderId: string, phone: string): Promise<{ checkoutRequestId: string }>;
    getStatus(orderId: string): Promise<PaymentStatus>;  // the reconciliation poll
  };
  orders: {
    lookup(reference: string, phone: string): Promise<Order>; // guest tracking
  };
  admin: AdminSurface;   // auth + membership required — never imported by the storefront
}

§05 Payment flow

The hard part: a payment that can only ever apply once.

STK Push has a notorious failure mode — the customer pays, but the callback never arrives. So two independent paths can confirm a payment, and an idempotency key guarantees neither can double-charge or oversell the last unit.

1
Customer → BFF
Create order. Items + phone posted; server reprices from the DB, computes VAT, writes the order as pending.
2
BFF → Edge fn
Record intent. A payment row is written with a unique idempotency_key before anything leaves the building.
3
Edge fn → Daraja
STK Push. Fired against the tenant’s own shortcode; the customer gets a PIN prompt on their phone.
4
Daraja → Callback
Path A — the callback. On success, the edge function applies the payment once, guarded by the idempotency key.
5
Customer → BFF
Path B — the poll. If no callback lands, the client polls status; the BFF queries Daraja and reconciles through the same guard.
6
BFF → Postgres
Settle atomically. In one locked transaction: mark paid, decrement variant stock, append an inventory_log — no oversell under concurrency.
7
Postgres → Customer
Confirm. Order flips to paid; receipt is issued with the VAT breakdown when the tenant’s toggle is on.

§06 Security model

Isolation lives at the database, not just the app.

Row-Level Security guards the client path. But callbacks and inventory writes run with the service role, which bypasses RLS entirely — so the privileged path carries and checks business_id in code. Two paths, two enforcement points, one boundary.

RLS — the client path

Every tenant table enables Row-Level Security. Authenticated writes require a matching business_members row; public reads see only published catalog rows.

Service-role discipline — the privileged path

Server code that bypasses RLS (callbacks, stock decrements) must stamp business_id from the resolved host and validate it. “RLS handles it” is treated as a bug.

Trust nothing from the client

Prices and totals are recomputed server-side from the database at checkout. The client proposes a cart; the server decides what it costs.

Secrets never touch the browser

Daraja passkeys are encrypted at rest and read only by edge functions under the service role. No tenant credential is ever serialized to a client.

§07 Roadmap to launch

Bizrah live in about two weeks — pending Safaricom.

The code is a two-week build for one focused engineer. The one thing outside our control is Safaricom’s production go-live approval, so it’s submitted on day one and we build against sandbox until it clears.
